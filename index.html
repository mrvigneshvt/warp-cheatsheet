Agent Profiles & Permissions

🛡

Safe & Cautious

Read-only operations. Always asks before executing commands. Ideal for production environments.

Read-Only Always Ask
⚖

Balanced

Agent decides which commands to run. Good balance of autonomy and safety for daily development.

Agent Decides Smart
🔥

YOLO Mode

Full autonomy. Runs commands without asking. Maximum speed, minimum friction. Use with caution.

Full Access Always Allow

⚡ Run Until Completion

Let the agent run multiple commands without pausing for approval between each one.

Ctrl+Shift+I
Warning: Only use in YOLO mode or when you trust the agent's plan. Monitor terminal output.

✅ Command Allowlist (Safe by Default)

These read-only / safe commands are typically allowed:

which, ls, grep, find, echo, cat, head, tail, wc, stat, pwd, date, uname, git status, git log, git branch, git diff

❌ Command Denylist (Blocked)

These dangerous commands are blocked by default:

wget, curl, rm, eval, mkfs, dd, shutdown, reboot
Tip: Customize allowlist/denylist with regex patterns in agent settings for project-specific control.

📜 AGENTS.md Structure & Best Practices

The AGENTS.md file (must be ALL CAPS) provides project-specific instructions to the agent:

# AGENTS.md ## Project Overview Brief description of the project and its purpose. ## Architecture Key architectural decisions and patterns. ## Coding Standards - Language conventions - Testing requirements - Code style guidelines ## Commands Common commands for building, testing, deploying. ## Important Notes Any gotchas, known issues, or special considerations.
Subdirectory Rules: Place AGENTS.md in subdirectories for directory-specific instructions. Subdirectory rules take precedence over root-level rules.

🌐 Global Rules

Set up global rules that apply across all projects:

# Location: ~/.warp/AGENTS.md (or ~/.agents/AGENTS.md) # These rules apply to ALL projects unless overridden # Use for personal coding preferences, common patterns, etc.
Best Practice: Use global rules for personal preferences (language, formatting style) and project rules for project-specific architecture and commands.

Planning & Execution

📋 /plan Workflow

  1. Create: Use /plan or describe your goal. The agent generates a structured plan with steps.
  2. Review: Examine the plan in the plan panel. Each step shows status, dependencies, and estimated effort.
  3. Edit: Modify steps, reorder, add/remove tasks before execution.
  4. Execute: Agent works through steps sequentially, marking each complete.
  5. Monitor: Watch real-time progress. Pause, skip, or modify steps mid-execution.

💾 Version History

Plans maintain version history. Revert to previous versions or compare changes between iterations.

Access: Plan panel > History tab > Select version to restore.

🔄 Plan Shortcuts

Ctrl+Alt+P
Toggle plan panel
Mid-run updates: Plans can be updated during execution. The agent adapts to changes without restarting.

🔁 Cross-Conversation Plan Reuse

Reference plans from other conversations using @plans context attachment:

# In any conversation, attach a previous plan: @plans [plan name or description] # The agent loads the plan context and can: # - Continue from where it left off # - Adapt the plan to the current project # - Compare with existing plans

Skills System

📜 Skill File Format

Skills are markdown files with YAML frontmatter:

--- name: my-skill description: What this skill does version: 1.0.0 --- # My Skill ## Instructions Step-by-step instructions for the agent... ## Examples Example usage and expected outputs...

📁 Project Skills Locations

Skills scoped to a specific project:

.agents/skills/ .warp/skills/ .claude/skills/ .cursor/skills/ .gemini/skills/

🌐 Global Skills Locations

Skills available across all projects:

~/.agents/skills/ ~/.warp/skills/ ~/.claude/skills/ ~/.cursor/skills/

💬 Argument Syntax

SyntaxDescriptionExample
$ARGUMENTSAll arguments as a single stringuse skill $ARGUMENTS
$ARGUMENTS[N]Nth argument (0-indexed)$ARGUMENTS[0] = first arg
$NShorthand for $ARGUMENTS[N]$0, $1, $2
$0First argumentPrimary input parameter

🚀 Invocation Methods

Natural Language: Just describe what you want in plain English. Warp matches your intent to available skills.

"Create a new React component with props"

Explicit Invocation: Use the slash command to call a specific skill:

/my-skill [arguments]

🛠 Creating Skills (4-Step Process)

  1. Define: Create a markdown file with YAML frontmatter (name, description, version).
  2. Instruct: Write clear, step-by-step instructions for the agent to follow.
  3. Place: Save in the appropriate skills directory (project or global).
  4. Test: Invoke the skill and verify the agent follows instructions correctly.
Pre-built Skills: Browse community skills at warpdotdev/oz-skills on GitHub.

Rules System

🌐 Global Rules

Apply to all projects. Set personal coding standards and preferences.

# Locations: ~/.warp/AGENTS.md ~/.agents/AGENTS.md

📁 Project Rules

Apply to a specific project. Define architecture, commands, and conventions.

# Location (project root): AGENTS.md # Must be ALL CAPS

📈 Precedence Order

Subdirectory AGENTS.md → Root AGENTS.md → Global AGENTS.md
Highest priority wins: More specific rules override general ones. Subdirectory rules take precedence over root, which takes precedence over global.

Accuracy Checklist

Before every agent interaction, verify these 6 items for maximum accuracy:

  • ✓
    Codebase is indexed — Verify indexing is complete and up-to-date in Settings > AI > Codebase Indexing.
  • ✓
    AGENTS.md exists — Project root has an AGENTS.md file with architecture, commands, and standards.
  • ✓
    Correct model selected — Model matches task complexity (Sonnet for daily, Opus for complex, Haiku for quick).
  • ✓
    Right profile active — Profile matches environment (Safe for prod, Balanced for dev, YOLO for sandbox).
  • ✓
    Relevant context attached — Files, blocks, or symbols referenced with @file, @symbol, or block navigation.
  • ✓
    Permissions configured — Command allowlist/denylist appropriate for the current task and environment.

Efficiency Patterns

🔬 Debug → Fix → Verify Loop

The fundamental debugging workflow for agent-driven development:

  1. Debug: Describe the bug. Agent analyzes error output, stack traces, and relevant code.
  2. Fix: Agent proposes and applies a fix. Review the diff in the code review panel.
  3. Verify: Run tests or reproduction steps. If still broken, loop back to Debug with new context.
Key: Always include error output as context. Use block attachment to show the failing command output.

🌱 Feature Development

Structured approach to building new features with the agent:

  1. Plan: Use /plan to create a step-by-step feature plan.
  2. Implement: Agent works through plan steps, creating files and modifying code.
  3. Review: Use code review panel to inspect all changes.
  4. Test: Agent runs tests and fixes any failures.

🔄 Refactoring Pattern

Safe refactoring with agent assistance:

  1. Scope: Define what to refactor and why. Attach relevant files.
  2. Plan: Agent creates a refactoring plan with before/after comparisons.
  3. Execute: Apply changes incrementally. Verify after each step.
  4. Validate: Run full test suite. Check for regressions.

📚 Learning a Codebase

Use the agent to understand unfamiliar projects:

  1. Overview: Ask the agent to explain the project structure and architecture.
  2. Deep Dive: Use @symbol to explore specific functions, classes, or modules.
  3. Flow Tracing: Ask the agent to trace execution paths for specific features.
  4. Document: Have the agent generate documentation or diagrams.

Multi-Repo Context

🌐 Cross-Repo Referencing

Work across multiple repositories in a single agent conversation:

  • 🌐
    Both repos must be indexed — Ensure codebase indexing is enabled for all repositories you want to reference.
  • 🌐
    Mention repo names explicitly — Use the repository name in your prompt so the agent knows which codebase to reference.
  • 🌐
    Use @file with repo path — Reference files across repos using full paths or repo-qualified paths.
  • 🌐
    Combine contexts — Attach blocks, files, and symbols from multiple repos in the same conversation.
# Example multi-repo prompts: "In the auth-service repo, how does the login flow work?" "Compare the User model in api-gateway with the one in user-service" "Update the shared types in common-lib to match the new API in backend"
Important: Cross-repo context only works when both repositories have been indexed. Check indexing status in Settings > AI > Codebase Indexing for each repo.
